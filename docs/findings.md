# 项目发现与结论

## 代码库基线（2026-07-11）

### FreeRTOS 配置
- `configUSE_PREEMPTION=1`, `configTICK_RATE_HZ=1000`
- 当前选用的 GCC ARM_CM4F FreeRTOS 移植层不使用 `configENABLE_FPU=0`。工具链参数为 `-mfpu=fpv4-sp-d16 -mfloat-abi=hard`；`port.c` 已启用 VFP 和浮点上下文惰性保存。该项已验证关闭。
- `configCHECK_FOR_STACK_OVERFLOW=2`
- `configMAX_PRIORITIES=56`，堆大小为 32 KB
- 已启用 CMSIS-RTOS v2 封装层

### 任务布局
| 任务 | 优先级 | 周期 | 栈大小 |
|------|----------|--------|-------|
| `controlTask` | 高 | TIM7 候选 1 kHz 直接通知 | 1536 B |
| `safetyTask` | 高 | 100 ms | 1024 B |
| `commTask` | 较高 | 2 ms | 2048 B |
| `imuTask` | 较高 | DRDY/10 ms | 1536 B |
| `monitorTask` | 低 | 500 ms | 1024 B；运行灯与栈水位，不再操作 ADC |

### 当前 `AppImuOutput` 标志
- `SENSOR_PRESENT`、`CALIBRATED`、`DATA_VALID`、`FILTER_INITIALIZED`
- `FILTER_CONVERGED`、`ACCEL_UPDATE_USED`、`VIBRATION_HIGH`

### IMU 数据与标志语义

| 字段/标志 | 含义 | 控制链用途 |
|---|---|---|
| `raw_sample` | 未修改的 QMI8658A 同步寄存器原始值 | 不进入控制，仅用于诊断和遥测 |
| `acceleration_mps2` | 估计器使用的当前机体系 SI 加速度 | 仅在处理后有效性条件满足时使用 |
| `angular_rate_rad_s` | 已扣除估计陀螺零偏的当前机体系角速度 | 仅在处理后有效性条件满足时使用 |
| `quaternion`、`euler_rad` | ESKF 姿态输出 | 需要 `DATA_VALID`、`FILTER_INITIALIZED`、`TIMESTAMP_VALID` 及对应姿态有效标志 |
| `SENSOR_PRESENT` | 传感器已检测并初始化 | 仅是诊断前提，不代表数据有效 |
| `CALIBRATED` | 启动静止标定已完成 | 处理后输出的必要条件 |
| `DATA_VALID` | 当前处理样本及估计器状态可用 | 控制和安全消费者的必要条件 |
| `FILTER_INITIALIZED` | ESKF 名义状态和误差状态已初始化 | 姿态输出的必要条件 |
| `FILTER_CONVERGED` | 已达到最低有效重力更新次数 | 就绪和诊断门槛 |
| `ACCEL_UPDATE_USED` | 当前加速度样本通过重力观测门控 | 用于诊断；单次拒绝不构成传感器故障 |
| `VIBRATION_HIGH` | 加速度模长表明振动或动态运动较强 | 诊断和降级输入 |
| `SENSOR_FAULT` | 当前策略下出现持久的传感器访问或数据故障 | 禁止消费处理后数据 |
| `DATA_STALE` | 没有足够新的有效样本 | 禁止消费处理后数据 |
| `TIMESTAMP_VALID` | 传感器时间推进正常 | 处理后消费者的必要条件 |
| `TILT_VALID` | 横滚角和俯仰角可观且有效 | 使用倾角时的必要条件 |
| `ABSOLUTE_YAW_VALID` | 绝对航向可观 | 当前仅使用六轴 IMU 的估计器中始终为假 |

M1 已增加名称明确的滤波加速度和滤波角速度字段，供控制与遥测使用；这些字段不会替代 `raw_sample`，也不会进入 ESKF 输入链路。

### M1 故障与恢复语义

| 状态 | 含义 | 安全行为 |
|---|---|---|
| `HEALTHY` | 当前样本和已初始化的有限估计器通过处理后数据门控 | 允许使用处理后的 IMU 输出 |
| `TRANSIENT_DEGRADED` | 单次传感器访问、时间戳或突变错误影响当前样本 | 禁止运动，但不永久锁存 |
| `PERSISTENT_SENSOR_FAULT` | 连续传感器或突变失败达到配置门槛 | 保持运动禁止，同时继续重试和上报心跳 |
| `ESTIMATOR_FAULT` | ESKF 更新产生无效内部状态，并尝试从标定零偏重新初始化 | 保持运动禁止，并要求连续稳定恢复样本 |
| `RECOVERING` | 传感器读取和估计器更新已经恢复，但尚未满足稳定样本门槛 | 恢复完成前保持 `DATA_VALID` 清零 |

任务心跳状态通过 `AppRuntimeSnapshot.critical_tasks_alive` 独立发布。当前只有重复的关键任务心跳丢失会永久锁存；IMU 降级仅设置 `motion_inhibited`，同时保持重试和恢复。

### M2 电机控制决策状态

- 当前硬件没有 MCU 可见的电机电流反馈。AT8236 的 `ISEN` 分流电阻和固定 `VREF` 只提供硬件电流斩波，未连接到 STM32 ADC，因此 M2 禁止建立伪电流环。
- 当前 SYSCLK 为 168 MHz、PWM 约为 20 kHz，`controlTask` 已切换到 TIM7 候选 1 kHz 软件节拍并待板测。仅保留 168 MHz 作为 MCU 计划运行点，PWM 和控制频率仍是实验候选值。
- LADRC 是基准控制器，但不是最终选定控制器。PI + 前馈和 PI + DOB 是 `m2_control_architecture_gate.md` 定义的对比候选方案。
- 首轮速度环候选为 1 kHz。资源审查确认 TIM6 已用于 HAL 时基，因此使用 TIM7 触发任务通知；最终采样率和闭环带宽需要系统辨识、延迟/噪声测量及离散鲁棒性证据。
- 正式确定控制架构前必须完成电流反馈硬件决策。如果增加电流采样，需要重新评审电流环时序和负载转矩观测方案。

### M2 G0 电机身份（2026-07-15）

- 已购电机为塔克 MC520P30_12V，12 V、1:30、1024 PPR 磁编码器 AB 相版本。
- 原厂参数为：空载转速 360 ± 20 rpm、额定转速 290 ± 20 rpm、额定电流 0.3 A、堵转电流 3.2 A、额定转矩 1.5 kg·cm、堵转转矩 4.5 kg·cm、额定功率 4.32 W、质量约 150 g；同时给出电感 4.45 mH、电阻 2.3 ± 0.5 Ω。
- 电机标称堵转电流 3.2 A，高于控制板约 2.2 A 的 AT8236 硬件斩波阈值。辨识时必须显式检测斩波介入区域，不能假设能够达到目录堵转点。
- 2026-07-18 MA 车轮人工向前三整圈的完整起止平台累计 `-371783` counts，平均绝对值 `123927.7 counts/rev`，相对 `1024 * 30 * 4 = 122880` 仅高约 0.85%，并排除 30720。固件按 1024 PPR、AB 相四倍频和 1:30 减速比配置为 122880 counts/wheel-rev；人工标记误差使该试验用于关闭倍率，不替代未来自动化精密标定。

### M2 G0 电池与机械参数（2026-07-15）

- 电池为 LB350E，采用 3 串 21700 电芯，容量 5000 mAh，标称电压 11.1 V，满电电压 12.6 V，推荐充电电流 2 A，最大持续放电电流 8 A，最大瞬时放电电流 16 A，质量约 230 g，尺寸约 75 × 64 × 22 mm；放电使用 T 型防反接插头，充电使用 DC5.5 × 2.1 接口。
- 规格图虽然标注“3C 电芯”，但整包参数表明确给出的最大持续放电电流是 8 A；安全设计按整包 8 A 限值执行，不按 `5 Ah * 3C = 15 A` 推算持续能力。
- 电池规格图声明具备过充、过流、过放、短路、过压保护和电芯安全阀，但没有给出 BMS 具体动作阈值、恢复条件和 16 A 允许持续时间；这些参数不能推测。
- 整车空载质量为 2.2 kg，最大额外载荷约 10 kg，最大总质量按约 12.2 kg 规划；轮径为 77 mm，底盘和主要机械结构为金属材质。
- 电机安装和出线方式与参考底盘及源码一致，M2 先沿用参考符号映射，不安排独立方向试验。MA 首次低功率开环斜坡必须顺带检查指令与速度符号；若不一致，在 BSP/配置层修正，其余三轮留到 M4 联调。
- 四台电机目录堵转电流合计 12.8 A，高于电池 8 A 持续能力；按每路约 2.2 A 的 AT8236 硬件斩波估算，四路理想总和仍约 8.8 A。G1 必须配置启动斜坡、PWM 上限和欠压/保护遥测，并通过单路到四路的分级试验确定实际边界。
- “金属材质、耐温高”只作为结构描述，不构成电机、驱动器或电池允许温度证据；量化热保护阈值仍需厂家资料和温升试验。

### G1.1 确定性控制时基结论

- TIM6 继续作为 HAL 1 kHz 时间基准；TIM7 使用 84 MHz 输入、`PSC = 83`、`ARR = 999` 和数值优先级 5，产生候选 1 kHz 控制节拍。SYSCLK/HCLK 保持 168 MHz。
- DWT 时间戳在 TIM7 向量入口采集，因此唤醒延迟包含 HAL 中断分派、编码器快照、FreeRTOS 通知和调度开销。编码器实际读取位于同一次 ISR 的 HAL 回调中，相对入口时间戳存在该 ISR 内固定顺序偏移，需在板上 WCET 中计入。
- 计数型通知只说明待处理次数，不能单独保证通知与单份“最新快照”一致。当前采用 16 槽带序号环，以消费前取得的通知数定位对应快照；后到 ISR 不会让当前任务误取未来快照。
- TIM7 的 UIF 会折叠多个未服务更新，不能只用相邻 IRQ 间隔检测漏周期。当前从启动基准累计无符号 DWT 间隔，用 64 位累计理论应到更新数和实际 ISR 服务数之差记录硬件漏周期；任务漏迭代和通知合并另行计数。
- DWT 32 位计数约 25.6 s 回绕，单次入口差使用无符号减法，再累计到 64 位。诊断成立的前提是相邻实际 TIM7 ISR 间隔小于一次 DWT 完整回绕；超过该边界已经属于严重调度失效。
- 软件离线验证已关闭通知—快照竞态、16 位编码器正反向回绕、序号/DWT 回绕、P99 门槛向上取整和 UIF 折叠反例。板上零丢周期、P99、最大抖动、WCET 和栈水位仍未确认。

### G1.2 MA 安全开环链路结论

- MA 开环状态机与 HAL 解耦，状态覆盖未使能、已使能、运行、停车、反转制动、可恢复禁止和 ESTOP 锁存。840 硬限幅、1 count/ms 斜坡、1 个反转 Brake 节拍和 500 ms 超时均由编译期配置约束。
- 超时停车具有失效保护优先级：进入超时减速后，`SET_PWM` 和 `STOP` 都不能清除超时/停零撤销使能门槛；只有到零进入 `DISARMED` 后显式 ARM 才能恢复。
- 可恢复 IMU 降级或单个关键心跳窗口缺失触发普通 Coast，不关闭 PWM 定时器；连续 3 个关键心跳窗口缺失、外部硬故障或 ESTOP 才调用 `BspMotor_EmergencyCoastAll()`。该接口停止 TIM1/TIM9/TIM12、清除输出通道并把引脚改为 GPIO 低电平，在线命令不能恢复。
- `controlTask` 是 MA 普通输出所有者，`safetyTask` 只执行安全覆盖。两者通过短任务级仲裁区形成全序，防止 Safety 已 Coast 后 Control 再按旧门状态写入 PWM；该区域不屏蔽 TIM7 ISR，也不包含等待或阻塞式 HAL 调用。
- MB、MC、MD 在每个正常输出周期都被强制 Coast。G1.2 提交完成时没有外部运动请求入口；G1.3 现已接入受严格解析、序号、有序队列和显式 ARM 约束的可配置调试串口调用者，烧录镜像只有收到有效命令后才可能进入 Drive。
- 软件验证已覆盖状态、时间和序号回绕及安全升级；板上实际 PWM 极性/波形、Brake/Coast、电机符号、故障注入、WCET、抖动和栈水位仍未确认。

### G1.3 临时命令与遥测结论

- 临时协议固定为大写 ASCII：`ARM <seq>`、`PWM <seq> <value>`、`STOP <seq>`、`ESTOP`、`STATUS`。LF 前最多 64 B，可带尾随 CR；数值溢出、±840 越界、控制字符、旧序号和超长行均被拒绝并分类计数。
- `ARM/PWM/STOP` 使用同一 32 位半区间单调序号。解析与序号提交采用两阶段语义：实时运动许可开放且深度 8 的有序命令队列成功入队后才提交；许可关闭或队列满都不提交并允许同序号重试。控制任务每个 TIM7 节拍最多消费 1 条，并用许可代际拒绝故障前遗留请求。
- 格式错误、旧序号、越界值和模拟队列满均不会调用电机状态机，因此不刷新 500 ms 超时。`ESTOP` 绕过普通队列，在既有执行器仲裁区内立即锁存故障并 Emergency Coast；只能系统复位恢复。
- 调试端口发送只使用 `HAL_UART_Transmit_IT()`。每端口深度 4 的 416 B 帧队列由单任务生产，完成 ISR 和任务恢复通过 C11 原子三态管理所有权；启动前先取得所有权再读取队首，关闭了完成 ISR 插入造成旧帧重复发送或槽覆盖的竞态。
- 20 ms 整数遥测覆盖控制时间戳/节拍、四路编码器计数、MA PWM、电池、关键心跳、运行就绪、安全与复位状态、IMU 年龄/健康、IRQ 抖动、唤醒延迟、最大 WCET、漏周期/截止期、许可前置拒绝和旧队列命令失效计数。全极值帧在 416 B 内保留至少 6 B，且低于 230400 8N1 的每周期持续字节预算。
- ADC 只由 `commTask` 按 20 ms 周期采样；`STATUS/ESTOP` 复用最近电池快照。ADC 轮询参数从 10 ms 收紧为 1 ms，仍需板上验证电压精度、异常超时和任务栈水位。
- G1.3 原始软件离线验证为 host 9/9、Debug/Release 66/66、7 个关键源文件严格告警与静态分析通过。当前 M2 板测镜像已把调试链临时路由到 USART2；持续收发、队列拥塞、错误恢复、ADC、MA 实际输出和 M1.3 联合验收仍待板测。

### M2 Type-C/USART2 板测路由

- `app_debug_uart_config.h` 是调试命令、遥测与端口统计的唯一选口位置；当前值为 `BSP_UART_ROS`，即 USART2。`commTask` 不再直接引用 `BSP_UART_TTL`。
- USART2 与 UART4 的 BSP 初始化、逐字节接收中断、异步发送队列和错误恢复均保留；本次只改变应用层调试链消费者，不修改 `.ioc`、PD5/PD6、PC10/PC11 或 230400/8N1 配置。
- 当前 USART2 仅运行现有大写 ASCII 板测协议。M5 正式启用 X-Protocol 前必须把调试配置迁回 `BSP_UART_TTL/UART4`，不得在 USART2 上同时自动探测两种协议。
- 2026-07-18 实机确认 H60 V3.6.0 的 USB-DEBUG/CH9102 对应 USART1，不能接收当前调试链；连接 USART2 PD5/PD6 的 USB-COM 后枚举为 COM6。ST-Link 3.28 V 单独供电、电机无动力且车轮悬空时，COM6 以 230400/8N1 在 2.2 s 收到 100 条遥测，`STATUS` 接受计数增至 1；三次关闭/重开均立即恢复 40—41 条/0.9 s，期间 `uwTick` 持续增长，UART 收发、协议和遥测错误计数均为 0。由此完成提交一的串口映射、双向通信和复开冒烟验收。

### Windows 串口采集与证据格式

- `capture_serial.ps1` 每次生成不可混淆的独立目录；`raw_uart.log` 保持串口原始字节，`telemetry.csv` 增加主机相对时间/UTC 后展开固件 40 个值字段，`commands.csv` 区分计划与实际发送时间，`metadata.json` 绑定固件提交、工具哈希、主机、端口和计数。CSV 是分析入口，原始日志仍是解析争议时的最终证据。
- COM 口和固件提交不能自动推断：Windows 端口号会随物理接口变化，当前仓库提交也不等于板上固件。工具因此把二者设为显式参数，并在元数据中另记仓库提交和脏状态，避免把“运行了哪个采集器”和“板上烧了哪个镜像”混为一谈。
- 默认只允许 `STATUS`，非状态命令需要单独解锁。这一限制防止无动力通信验收误发运动指令，但不能取代 M2 后续单电机实验的人工安全检查、固件运动许可和命令代际保护。
- 2026-07-18 COM6 实采证明原始文件字节数、元数据计数和 CSV 行数可以闭合：5 s 为 33,060 B/228 行，解析错误、非遥测行和尾部残片均为 0，板端时间严格单调，8 个 `C` 组错误字段全程为 0。

### G2/G3 无动力试验准备

- 首次运动计划必须解决 500 ms 命令超时：只发一次 PWM 后长时间等待会触发不可取消的超时降零。计划生成器因此以不超过 400 ms 的间隔使用连续新序号重发同一 PWM，最后显式 STOP；正反方向分开，避免首次运动就跨零换向。
- 固件 840 上限是 G1 风险边界，不是首次上电建议值。提交三额外把首动计划生成限制为 200，并仍要求操作者在接入动力前确认本次具体 PWM；该软件临时上限不修改固件，也不代表已经证明 200 对实机安全。
- 现有 50 Hz 遥测可从累计编码器计数计算相邻帧 counts/s，并读取累计最大抖动、WCET、漏周期和截止期；它不能观察每个 1 kHz 样本的完整分布，也没有脉冲周期。故当前只做时序初查和计数率分析，G3 仍为未就绪，禁止提前换算 RPM 或宣称 M/T 已验证。

### IMU 陈旧样本与安全门闭环

- 2026-07-15 详细注释审查曾发现：持续无新样本只会设置 `DATA_STALE`、清除 `DATA_VALID`，却不会同步把健康等级改为降级。2026-07-18 已将陈旧边界统一为 `>= 20 ms`，并在 IMU 层关闭健康分级和稳定恢复缺口。
- `AppImu_IsMotionUsable(output, now_ms)` 现集中检查健康、有效标志、ESKF 收敛和实时年龄，避免消费者只读取可能滞后的 `sample_age_ms`。持续 `BSP_BUSY`、时间戳重复、超时边界、稳定恢复和退避快照刷新已有主机测试。
- 统一判据现已接入启动就绪、安全任务、通信普通命令前置门和控制任务最终执行器复核。`runtime_ready` 只有在关键任务心跳完整且 ESKF 收敛后才单向锁存；动态降级仍立即关闭运动许可。
- `ARM/PWM/STOP` 在许可关闭时不入队也不提交协议序号，恢复后可用同一序号重试；`STATUS/ESTOP` 始终独立分流。许可下降沿递增代际，故障前已排队但尚未执行的普通命令不会在恢复后补执行。
- 控制任务在实际写 PWM 前同时复核实时 IMU 可用性、运行就绪、运动禁止、锁存故障和请求代际。安全任务、IMU 发布、通信准入和控制提交对许可更新使用同一种不屏蔽 TIM7 ISR 的调度仲裁边界。
- 带许可代际的命令队列元素从 12 B 增至 16 B，深度 8 因此从既有 FreeRTOS 固定堆中多占 32 B；这部分不会增加链接器报告的静态 RAM 总量，但会减少同等大小的运行时剩余堆，板测仍需记录实际最小堆余量与各任务栈水位。

### IMU 陈旧判定与快照年龄竞态（2026-07-18）

- 原实现使用 `sample_age_ms > 20 ms`，使恰好 20 ms 时仍可能保留有效状态；现已统一为 `>= 20 ms`，并在首次陈旧时进入明确的传感器降级与稳定恢复流程。
- 仅检查快照内的 `sample_age_ms` 仍存在发布延迟：控制任务执行时，真实年龄可能已经越过门槛而快照年龄尚未刷新。因此公共运动判据必须接收当前 `now_ms`，以 `now_ms - last_good_tick_ms` 重新核验，不允许各消费者自行复制一套阈值逻辑。
- 恢复阶段的重复时间戳虽然不属于总线读错，但会破坏“连续新样本”条件，必须清零稳定计数；普通数据未就绪和退避轮询不能每次都清零，否则当 IMU 任务轮询快于芯片 ODR 时将永远无法恢复。
- 正常健康运行与故障恢复使用不同严格度：单帧加速度观测拒绝或高振动在正常运动中可能合理，不能单独触发停车；它们只应作为后续严格恢复样本资格的一部分。

### IMU 突变基线与估计器时间基准（2026-07-18）

- 原实现把被拒绝的尖峰向量写成下一帧比较基线，导致单点异常在“正常→尖峰”和“尖峰→正常”两个边沿各被拒绝一次。突变基线现在只由真正接受的样本更新。
- 仅修复向量基线仍不够：原始时间戳在尖峰被拒绝时已经前进，若下一次 ESKF 仍只使用相邻原始帧步长，会漏掉尖峰占用的一个采样周期。现改为独立累计尚未送入 ESKF 的时间步，成功更新或估计器按当前样本重建后才清零。
- 估计器累计时间按实际空缺推进，但上限仍为 50 ms；长间隔不能恢复成精确轨迹，只能以有界预测进入后续稳定恢复。因此时间戳大跳变仍属于降级事件，不能因使用累计时间而视为正常丢样。
- 恢复资格必须比健康运行更严格。方向创新拒绝和高振动在健康运动中只影响当前重力校正，不改变健康等级；在恢复阶段则会清零连续稳定计数，防止动态或不可信重力样本过早解除安全禁止。
