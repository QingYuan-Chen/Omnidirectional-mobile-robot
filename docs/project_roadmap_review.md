# 项目总体路线图审阅（Roadmap Review）

> 审阅人: Claude（2026-07-11），M2 控制架构部分于 2026-07-14 修订。本文档整合 `CubeMX工程配置计划.md`、`FreeRTOS框架规划.md`、`task_plan.md`、`plan_review_and_directives.md` 四份文档，给出全项目层面的路线图、缺口与修正建议。
> 分工：本文档管"到终点的全程规划"；`plan_review_and_directives.md` 管"当前三阶段怎么干"。

## 1. 项目终点定义（此前所有文档都没有明确写出）

**最终交付物**：R750-MEC 四麦轮底盘下位机固件，能够：

1. 通过 USART2 以 X-Protocol（230400）接收 ROS2 上位机速度指令 `vx/vy/wz`，执行麦轮逆解 + 四轮速度闭环；
2. 周期回传轮速/机体速度、IMU（四元数/欧拉角/原始值）、电池电压、系统状态与错误码；
3. 具备完整安全链：命令超时受控停车、低电压三级策略、故障降级/锁存、IWDG 兜底；
4. UART4 保留人类可读调试台。

**最终验收**：与 `参考/7 ROS2源码资料` 中的上位机驱动节点实机联调，速度指令闭环执行、里程计/IMU 话题数据正确。

## 3. 总路线图（M0–M6）

每个里程碑必须有板上验收记录进 `progress.md` 后才算关闭。

- **M0 前置修复**（= directives 的 P0）：git 初始化、UART RX 停摆缺陷、IMU 故障恢复策略重设计。
- **M1 IMU 数据质量**（= Phase 1）：验收 = 干扰注入后自动恢复，不永久锁存。
- **M2 控制架构决策门 + 单电机闭环**（= Phase 2）：先完成确定性时基、开环辨识、测速评审、电流反馈硬件决策以及 PI + 前馈/LADRC/PI + DOB 统一比较，再按决策记录完成 MA 闭环；拔串口 500 ms 必须受控停车。
- **M3 人机交互**（= Phase 3）：验收 = 按键启停/急停、LED 状态机、非阻塞蜂鸣器、监控上报全通。
- **M4 四轮闭环 + 麦轮运动学 + 里程计**（新增，需在 M3 后立项）：
  - 复用 M2 决策后的统一电机控制器接口实例化四份；
  - `mecanum_kinematics` 纯函数模块（无 HAL 依赖）：`vx/vy/wz ↔ 四轮转速` 正逆解；
  - 里程计：下位机只回传四轮实际转速与机体速度，**位姿积分放上位机**（避免下位机累积漂移且与 ROS2 odom 习惯一致）；
  - 验收 = 手推小车回传速度符号/量级正确；给定 vx/vy/wz 空载轮速正确（含平移+自转组合）。
- **M5 X-Protocol 与 ROS2 联调**（新增）：
  - 帧 0x50/0x10/0x51/0x54 兼容原厂，新增 0x11/0x12；实现前先读 `参考/7 ROS2源码资料` 的上位机解析代码，**以上位机现有实现为兼容基准**，字段/字节序/校验不确定处对照源码确认后写进协议规格；
  - 命令超时与 M2 超时机制合并，不做两套；
  - 验收 = 上位机节点实测收发正常，错误帧注入不崩溃。
- **M6 生产加固**（新增）：IWDG（~1s，仅 Safety 在全心跳健康时喂）、UART2 改 DMA+IDLE、低电压三级策略（11.19V 提示 / 10.86V 强提示 / 10.50V 停车）、全任务栈余量与 CPU 占用实测收敛（目标最坏工况余量 ≥25% 且 ≥256B）、固件版本号进 0x12 状态帧。

## 4. 三项跨领域建议（当前规划体系的真正缺口）

### 4.1 坐标系与符号约定文档（M4 前必须存在）

麦轮项目最大的坑不是算法而是符号。建议新增 `docs/conventions.md`，一页纸钉死：

- 机体系遵循 REP-103：x 前、y 左、z 上，wz 逆时针为正；
- MA/MB/MC/MD 与左前/右前/左后/右后的对应关系（对照 H60 原理图确认，不许猜）;
- "电机正转"的统一定义 = 该轮使机器人前进的方向，编码器计数在 bsp 层统一成该约定（M2 的 2.0b 已要求单轮标定，M4 扩展到四轮）；
- 单位统一表：内部控制用 rad/s 还是 RPM、协议帧用什么单位、换算常数只在 `robot_config.h` 出现一次。

### 4.2 host 端单元测试（成本低、回报高）

`FreeRTOS框架规划.md` 已规定运动学、协议和控制器不依赖 HAL——M2 起建立桌面测试目标，覆盖速度估计、PI + 前馈、LADRC、PI + DOB、mecanum_kinematics 和 xprotocol 解析器。麦轮逆解和协议解析是纯函数，板上调错的时间远大于写测试的时间。

### 4.3 文档一致性修正（本次审阅发现的过时项）

`CubeMX工程配置计划.md` 多处已被代码超越，建议在文首加"历史文档，现状以代码与 FreeRTOS框架规划.md 为准"声明，或直接更新这几处：

| 文档说法 | 实际现状 |
|---|---|
| 时钟树"需用 H60 原理图确认晶振后固定" | `.ioc` 已固定 HSE=8MHz → SYSCLK 168MHz（若从未对照原理图核实过 8MHz，M0 顺手核一次并记录） |
| IMU ODR 建议 500Hz | 实配 CTRL2/CTRL3 为 224.2Hz，`robot_config.h` 与之一致 |
| 控制周期 20ms/50Hz | 当前软件为 10ms/100Hz 占位；M2 决策门从 1kHz 候选开始测量，最终值未冻结 |
| 任务表 MotorTask/CommRx/CommTx… | 实际为 control/safety/comm/imu/monitor 五任务（FreeRTOS框架规划.md 为准） |
| UART 分工"UART4 可选" | 钉死：USART2 = ROS X-Protocol，UART4 = 人类调试台（M2 起使用） |

## 5. 风险清单（按优先级）

1. **麦轮符号/轮序错误**（M4）——用 4.1 的约定文档 + 逐轮开环标定压制；
2. **IMU 故障策略过激**（M0/M1）——已在 directives P0.3 处理；
3. **CPU/中断余量与控制抖动未实测**（M2 前置）——M2 G1 就记录 1kHz 候选控制节拍下的 WCET、抖动和丢周期；M4 再用全通信负载复测，不能等 M6；
4. **低速编码器量化噪声**（M2）——必须比较 M 法、T 法和融合 M/T 法，并把滤波相位延迟纳入闭环评审，不能只加普通低通；
5. **协议兼容性想当然**（M5）——必须以上位机源码为基准，不凭记忆写帧格式。

## 6. 排期原则

- 严格串行 M0→M6，每个里程碑关闭前不开下一个（M4 内部四轮标定可与运动学模块并行）；
- 每个里程碑一次板上验收 + progress.md 记录 + git tag（`m1-imu`、`m2-motor-control`…）；
- 与上位机的联调放在 M5 就做，不要拖到项目最后才第一次对接。
